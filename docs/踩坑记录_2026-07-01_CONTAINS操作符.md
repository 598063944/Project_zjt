# 2026-07-01: [错误] 对象查询 CONTAINS 操作符错误处理

## 问题分类
[错误] - 明显的错误决策或实现

## 问题标题
对象查询筛选条件"包含"(CONTAINS) 获取不到数据

## 问题描述
用户在"设置-CRM设置-对象查询"中添加筛选条件"产品 包含 13.012"，设置完成后刷新数据仍获取不到预期结果。

## 根因分析

### 三个核心问题

**问题1**: L722-724 contains 操作符处理错误
- 获取客户端过滤列表 `c_lst` 但没有使用它
- 直接将条件 append 到 `api_filters`（发送给 CRM API）
- 试图发送 `'CONTAINS'` 操作符给 CRM API，但 CRM API 不支持

**问题2**: CRM API 不支持 CONTAINS 操作符
- CRM API 支持的操作符: EQ, LT, GT, GTE, LTE, N, LIKE, NLIKE, IS, ISN, IN, NIN, BETWEEN, NBETWEEN, STARTWITH, ENDWITH
- CONTAINS（数组包含判断）语义与 LIKE（字符串模糊匹配）完全不同
- contains/not_contains 应该在**客户端**进行内存过滤

**问题3**: 代码有重复的 elif 块
- L738-741 有两个 `elif op in op_map:` 块
- 第一个块只获取 crm_op 没有处理，第二个块才是真正的处理

### 影响范围
- 所有使用"包含"操作符的对象查询筛选
- 受影响字段: 产品、关联对象、多选字段等支持数组/包含判断的字段类型

## 解决方案

### 修改内容

**文件**: object_query.py  
**函数**: _build_api_filters_from_settings()  
**范围**: L720-741

### 修改逻辑

```python
# 修改前（错误）
elif op == 'contains':
    c_lst = getattr(self, '_obj_query_client_filters', [])
    api_filters.append({'field_name': api_field, 'operator': 'CONTAINS', 'field_values': [value]})  # ❌ 错！

# 修改后（正确）
elif op == 'contains':
    # CRM API: CONTAINS 无效，退回到客户端过滤
    c_lst = getattr(self, '_obj_query_client_filters', [])
    c_lst.append(cond)  # ✅ 添加到客户端过滤
    self._obj_query_client_filters = c_lst
```

### 设计理由

1. **API 端**: 不发送任何 contains 条件，直接获取**全量**数据
2. **客户端**: 在内存中逐行判断字段值是否包含搜索词
3. **已有支持**: 客户端过滤逻辑已在 L1368 正确实现

```python
# 客户端过滤实现（L1368）
if op == 'contains' and v not in cell_val:
```

## 经验教训

### 🎯 主要教训
1. **API 操作符的可用性**: 不要假设所有操作符都支持，需要先确认 API 文档
2. **API vs 客户端的权衡**: 某些过滤逻辑无法用 API 表达，应回退到客户端处理
3. **重复代码是隐形 bug**: 两个相似的 elif 块容易导致逻辑混乱

### 💡 最佳实践
- 构建 API 过滤条件时，先检查目标操作符是否被 API 支持
- 如果操作符在 API 不支持，主动添加到客户端过滤列表，而非盲目发送
- 保持单一责任：API 端处理可以表达的条件，客户端处理 API 无法表达的条件

### 🔍 排查建议
- 遇到"设置条件后获取不到数据"问题时，检查：
  1. 筛选条件是否被正确收集（在 UI 面板上）
  2. 条件是否被正确发送给 API（查看日志/网络请求）
  3. API 是否支持这个操作符（查看 API 文档）
  4. 是否应该回退到客户端过滤
