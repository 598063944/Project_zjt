## 根因分析

### 主因：QSS 样式表 padding 挤占复选框列

`custom_report.py` 新版 `ReportListPage._setup_ui()` 中：

```python
self._table.setStyleSheet("""
    QTableWidget::item { padding: 10px 12px; ... }
    QHeaderView::section { padding: 10px 12px; ... }
""")
```

这 20px（上下）的 padding 把 28px 宽的复选框列（列0）的实际可用空间挤没了。

**对比**：订单转合同（`order_to_contract.py`）的 `crm_table` 完全没有设置 QSS 样式表，Qt 默认渲染，复选框正常显示。两个表格的复选框创建逻辑完全一样。

### 次因：`TableRowCheckBox` 未导入

新版 `ReportListPage` 的 3 个方法中使用了 `w.findChild(TableRowCheckBox)` 但缺少 `from common import TableRowCheckBox`，导致 `NameError` 崩溃。旧版 `ReportListPage` 在对应方法中正确包含了这个导入。

## 复利方法（核心经验）

1. **QSS 样式表一旦设置了 `QTableWidget::item` 和 `QHeaderView::section` 的 padding，会把所有列的 cell 内容向内挤，复选框列（宽度仅 28px）直接不可见。** 以后给表格加样式时，要么不加 padding，要么对 col 0 单独设置 `padding: 0`。

2. **当两个 UI 组件表现不一致且代码逻辑相同时，差异一定在样式（QSS）或配置，不需要反复搜索逻辑代码。**

3. **类内部方法中如果用了另一个模块的类，务必在该方法内部 `from common import TableRowCheckBox`**——不要依赖模块级 import（否则重定义同名类时可能被覆盖）。

4. **遇到搜索循环时应该立刻停下来形成结论，而不是继续遍历同一个代码库。** 用户两次提醒"循环了吗"是有效的纠正信号。
