# 对象查询"包含"操作符问题排查与修复

**报告日期**: 2026-07-01  
**问题**: 设置-CRM设置-对象查询中，筛选条件"产品 包含 13.012"获取不到数据

---

## 问题诊断

### 【症状】
用户在对象查询设置中添加筛选条件：
- 字段：产品
- 操作符：包含
- 值：13.012

设置完成后，刷新数据仍获取不到预期结果。

### 【根因分析】

经代码审查，发现 `object_query.py` 中的筛选条件处理存在**三个关键问题**：

#### 问题1：contains 操作符处理错误（最严重）
**位置**: `object_query.py` L722-724（修复前）

```python
# ❌ 错误代码
elif op == 'contains':
    # CONTAINS = Array 包含（多选/人员字段）
    c_lst = getattr(self, '_obj_query_client_filters', [])
    api_filters.append({'field_name': api_field, 'operator': 'CONTAINS', 'field_values': [value]})
```

**问题**:
1. 代码获取了 `c_lst`（客户端过滤列表）但**没有使用它**
2. 直接将条件 append 到 `api_filters`（API 过滤列表）
3. 发送了错误的操作符 `'CONTAINS'` 给 CRM API
4. CRM API **不支持 CONTAINS 操作符**（只支持：EQ, LT, GT, GTE, LTE, N, LIKE, NLIKE, IS, ISN, IN, NIN, BETWEEN, NBETWEEN, STARTWITH, ENDWITH）

**结果**: 
- API 调用返回错误或无效结果
- 用户获取不到任何数据

#### 问题2：contains 不应走 API 路径
**CRM API 操作符映射**（L666-667）:
```python
'contains': 'LIKE',  # ← 这个映射没有被使用，因为 contains 在前面被特殊处理了
```

**根本原因**: 
- LIKE 是字符串模糊匹配（%value%），用于字符字段
- CONTAINS 是数组包含判断，用于多选/关联字段
- 这两个操作语义完全不同

**正确做法**: 
- `contains` 和 `not_contains` 无法通过 CRM API 的原生操作符实现
- 应该在**客户端**进行内存过滤（已有实现逻辑在 L1368）

#### 问题3：重复的 elif 条件块
**位置**: `object_query.py` L738-741（修复前）

```python
# ❌ 重复代码
elif op in op_map:
    crm_op = op_map[op]
elif op in op_map:    # 重复！
    crm_op = op_map[op]
    # 处理逻辑...
```

第一个块获取了 `crm_op` 后没有任何操作，第二个块才是真正的处理。这导致代码逻辑混乱。

## 修复方案

### 【修改内容】

**文件**: `object_query.py`

**修改范围**: L720-741（`_build_api_filters_from_settings` 方法内）

#### 修复前
```python
elif op == 'contains':
    # CONTAINS = Array 包含（多选/人员字段）
    c_lst = getattr(self, '_obj_query_client_filters', [])
    api_filters.append({'field_name': api_field, 'operator': 'CONTAINS', 'field_values': [value]})
elif op == 'not_contains':
    c_lst = getattr(self, '_obj_query_client_filters', [])
    c_lst.append(cond); self._obj_query_client_filters = c_lst
elif op in op_map:
    crm_op = op_map[op]
elif op in op_map:
    crm_op = op_map[op]
    # 日期类操作符：尝试将值转为 Unix 毫秒时间戳
    if op in ('lt', 'gt', 'lte', 'gte', 'date_before', 'date_after', 'eq', 'ne'):
        ts_value = self._to_timestamp_ms(value)
        api_filters.append({'field_name': api_field, 'operator': crm_op, 'field_values': [ts_value]})
    else:
        api_filters.append({'field_name': api_field, 'operator': crm_op, 'field_values': [value]})
else:
    skipped_conditions.append(field_name)
```

#### 修复后
```python
elif op == 'contains':
    # CRM API: CONTAINS 无效，退回到客户端过滤
    c_lst = getattr(self, '_obj_query_client_filters', [])
    c_lst.append(cond)
    self._obj_query_client_filters = c_lst
elif op == 'not_contains':
    # CRM API 没有"不包含"的原生操作符，退回到客户端过滤
    c_lst = getattr(self, '_obj_query_client_filters', [])
    c_lst.append(cond)
    self._obj_query_client_filters = c_lst
elif op in op_map:
    crm_op = op_map[op]
    # 日期类操作符：尝试将值转为 Unix 毫秒时间戳
    if op in ('lt', 'gt', 'lte', 'gte', 'date_before', 'date_after', 'eq', 'ne'):
        ts_value = self._to_timestamp_ms(value)
        api_filters.append({'field_name': api_field, 'operator': crm_op, 'field_values': [ts_value]})
    else:
        api_filters.append({'field_name': api_field, 'operator': crm_op, 'field_values': [value]})
else:
    skipped_conditions.append(field_name)
```

### 【修改要点】

1. ✅ **contains**: 改为添加到 `_obj_query_client_filters`（客户端过滤列表）
2. ✅ **not_contains**: 确保正确追加条件到客户端过滤列表  
3. ✅ **去重**: 删除重复的 `elif op in op_map` 块
4. ✅ **注释**: 明确说明为什么要退回到客户端过滤

## 工作流程验证

### 修复后的工作流程

```
用户设置：产品 包含 13.012
    ↓
1. API 过滤阶段
   - contains 操作符检测 → 添加到客户端过滤列表
   - 不向 CRM API 发送任何 contains 条件
   - API 返回**全量**产品数据
   ↓
2. 数据到达后的客户端过滤
   - 遍历所有返回的产品记录
   - 对每条记录的"产品"字段：
     * 转换为小写字符串
     * 检查是否包含"13.012"（也转小写）
     * 匹配则保留，不匹配则过滤掉
   ↓
3. 表格展示
   - 只显示包含"13.012"的产品记录
```

### 客户端过滤实现验证

客户端过滤逻辑已在 `object_query.py` L1368 正确实现：

```python
if op == 'contains' and v not in cell_val: 
    match = False
elif op == 'not_contains' and v in cell_val: 
    match = False
```

说明：
- `v` = 用户输入的值（"13.012"）
- `cell_val` = 数据行中该字段的值（转小写）
- 如果值**不**包含在字段值中，则过滤掉（match=False）
