**根因**：SalesOrderObj 有 400+ 个 CRM 字段，每列映射为 `LONGTEXT`。在 MySQL COMPACT 格式下每列前缀占用 768 字节，DYNAMIC 下占用 20 字节。

ALTER TABLE ADD COLUMN LONGTEXT 在列数过多时会抛出 ER_TOO_BIG_ROWSIZE (1118)。



**修改**：

  - 捕获 "Row size too large" 或 "1118" 错误

  - 设置 `SET SESSION innodb_strict_mode=OFF` 后以 force_recreate=True 重试

  - innodb_strict_mode=OFF 后 DDL 允许超宽表结构创建（LONGTEXT 数据存行外，不影响 DML）

## Part 2：复选框崩溃修复



### 症状



点击列表行复选框 → 程序闪退，控制台 "Unhandled Python exception"。



### 根因



_update_header_checkbox_state() 调用了不存在的方法：



`python

self._table_header.setCheckState(True)       # AttributeError!

self._table_header.setCheckState(False)      # AttributeError!

self._table_header.setPartiallyChecked(True) # AttributeError!

`



CheckBoxHeader（common.py:3043）实际只有 set_check_state(state)，接受 Qt.CheckState 枚举，且不发射 	oggled 信号。



由于 PyQt6 信号内异常被 Qt 事件循环吞掉，没有 traceback，直接闪退。



### 崩溃链



`

用户点复选框 → toggled → _on_row_checkbox_toggled

  → _update_header_checkbox_state

    → _table_header.setCheckState(True)  ← AttributeError → 闪退

`



### 修复（4处）



| 行号 | 原代码 | 修复为 |

|------|--------|--------|

| 26593 | setCheckState(False) | set_check_state(Qt.CheckState.Unchecked) |

| 26598 | setCheckState(True) | set_check_state(Qt.CheckState.Checked) |

| 26600 | setCheckState(False) | set_check_state(Qt.CheckState.Unchecked) |

| 26602 | setPartiallyChecked(True) | set_check_state(Qt.CheckState.PartiallyChecked) |



### 连带风险



如果 setCheckState 存在（camelCase），它会发射 	oggled 信号，导致：

_on_header_select_all_toggled → 遍历 setChecked → _update_header_checkbox_state → setCheckState → toggled → ...

形成无限递归栈溢出。



set_check_state() 只更新视图不发射信号，彻底避免此问题。



### 调试经验



1. **PyQt6 信号异常静默崩溃**：信号处理中的异常不打印 traceback，需加 try/except + 日志调试

2. **PyQt6 方法名是 PEP8 下划线**：不是 C++ camelCase。set_check_state 不是 setCheckState

3. **调用外部类前确认 API**：不要猜测方法名

4. **f-string 避免反斜杠**：提前用变量提取值

#### 4. Row size too large：主动设 innodb_strict_mode=OFF



**问题**：400+ 列的 CRM 对象（如 `NewOpportunityObj`、`SalesOrderObj`）建 source 表时报错：

```

ERROR: [ReportDatabase] 建 source 表失败 sr_NewOpportunityObj: 

(1118, 'Row size too large. The maximum row size for the used table type, 

not counting BLOBs, is 8126.')

```



**根因分析**：

- 每个 LONGTEXT 列在 InnoDB 行内占 20 字节指针，400 列 × 20 = 8000 字节，加上 `_id` VARCHAR(128) 约 386 字节（utf8mb4）、`_sync_time` DATETIME 5 字节、`_hash` VARCHAR(32) 约 98 字节，总计约 8500 字节，超过 8126 限制

- 旧代码先尝试正常创建，报错后才设 `innodb_strict_mode=OFF` 重试——但 ERROR 日志已经打出，干扰判断

- 且递归调用 `ensure_source_table` 通过 `_get_conn()` 复用同一连接，`SET SESSION` 变量可以保留，逻辑正确但首次 ERROR 日志误导



**修复方法**：

1. **主动预防**：在 `ensure_source_table` 开头判断 `col_count > 50`（预计可能超宽），**提前**执行 `SET SESSION innodb_strict_mode=OFF`，避免首次报错

2. **日志分级**：非超宽表的其他错误才打 `logger.error()`；超宽表重试成功只打 `logger.info()`

3. **新增计数日志**：建表成功后输出 `新增 X 列 (共 Y 字段)`，便于确认列数

4. **代码结构优化**：`is_row_too_large` 分支内不再打 ERROR（重试成功是正常路径），仅在重试也失败时才 ERROR

5. **保留两级兜底**：非超宽表错误 → 删表重建（兜底1）；超宽表 → 关 strict mode 重建（兜底2）



**涉及文件**：`custom_report.py` — `ReportDatabase.ensure_source_table`（L24856-24923）

#### 5. (0, '') 拼表错误修复：连接检测 + 空 SQL 防护



**问题**：批量刷新时偶发 `(0, '')` 错误（PyMySQL 错误码 0、空消息），导致拼表失败但无可排查的错误信息。



**根因分析**：

- PyMySQL 连接断开（MySQL 超时/网络抖动/服务重启）后，`conn.open` 属性有延迟不会立即变 False

- 对已断开的连接执行 `cur.execute()`，PyMySQL 抛出 `InterfaceError(0, '')` 或 `OperationalError(0, '')`

- 旧代码 `_get_conn()` 只检查 `self._conn.open` 属性，无主动探活机制

- `select_sql` 可能为空（边界情况），直接拼接 `CREATE TABLE AS {空SQL}` 导致 MySQL 语法错误



**修复方法**：

1. **`execute()` 方法增强**（L24783-24804）：

   - **空 SQL 防护**：`if not sql or not sql.strip():` 提前返回 None + 日志

   - **连接探活**：`conn.ping(reconnect=True)` 主动检测连接，断开时自动重连

   - **重连兜底**：ping 失败时 `self._conn = None` 强制 `_get_conn()` 创建新连接

   - **错误日志增强**：SQL 截断从 200 扩到 500 字符

   - **连接不可用日志**：`_get_conn()` 返回 None 时明确记录

2. **`create_result_table_as()` 增强**（L25119-25152）：

   - **空 SQL 断言**：`select_sql` 为空时抛 `ValueError("拼表 SQL 为空，无法创建结果表")`

   - **上下文日志**：错误日志含表名、SQL 长度、前 800 字符

   - **成功日志**：结果表创建成功后输出 `结果表创建成功: {table}`

#### 6. stderr 双重输出修复（ERROR: [SQLBuilder] 噪声）



**问题**：控制台出现大量 `ERROR: [SQLBuilder]` / `ERROR: [CompositeID]` 消息，且每条重复两次（一次带 ERROR 前缀，一次不带）。这些是调试日志，不是真正的错误，严重干扰真实错误排查。



**根因分析**（三层因果链）：

1. **直接原因**：`JoinSQLBuilder._build_composite_id()` 和 `_build_composite_id_column()` 中有大量 `print(f"[SQLBuilder]...", file=sys.stderr)` 调试输出（共 15+ 处）

**修复方法**：

1. **`Project_zjt_3.5.8.py` `setup_logging_to_file()`**（L3070-3082）：

   - 安装 `_StderrLogWriter` 后，遍历 root logger 的所有 handler

   - 将指向旧（被包装的）stderr 的 StreamHandler 重定向到 `_real_stderr`（原始未包装的 stderr）

   - 阻断日志→stderr→`_StderrLogWriter`→ERROR 的循环

2. **`custom_report.py` 调试输出清理**：

   - `JoinSQLBuilder._build_composite_id()`：全部 `print(..., file=sys.stderr)` → `logger.debug()`

   - `ReportRefreshWorker._build_composite_id_column()`：同上，去除 `import sys` 和 `traceback.print_exc()`

- `_collect_from_ui()` 通过 `_to_crm_api_name(c['source_object'])` 读取，并写入 `FieldColumn(source_object_api=src_api)`

- `FieldColumn` dataclass（L2571-2592）定义的字段名是 `source_object_api`（L2575），无 `source_object` 字段

- `_deserialize_report()`（L2719）加载 JSON 时执行 `FieldColumn(**c)`——如果旧数据 dict 中含 `source_object` 键，Python dataclass 默认 `__init__` 会抛出 `TypeError: unexpected keyword argument 'source_object'`

- 该异常被 `_load()` 的 `except Exception` 捕获，**整个报表加载失败**（从缓存中消失）

- 即使加载成功（dict 中只有 `source_object_api` 或两者皆无），`_load_to_ui` 中 `c.get('source_object_api', '')` 对含 `source_object` 的旧数据返回空字符串，导致 `_normalize_field_key("", field)` 跳过翻译，字段匹配失败



**修复方法**（4 层防护）：

1. **`_deserialize_report()` 加载时自动迁移**：检测 dict 中是否有 `source_object` 且无 `source_object_api`，是则 `c['source_object_api'] = c.pop('source_object')`，再传给 `FieldColumn(**c)`

2. **`ReportDefinition.__post_init__()` 构造时同样处理**：report 对象直接构造时（绕过 `_deserialize_report`）也能正确迁移

3. **`_load_to_ui` 画布卡片段**：`src_api = c.get('source_object_api', '') or c.get('source_object', '')` 双键回退

4. **`_load_to_ui` 字段面板段**：同上，`col.get('source_object_api', '') or col.get('source_object', '')`

### 2026-06-16: 日期成分提取（年/月/周/季度）批量刷新不生效 — 踩坑复利 ⚠️



#### 踩坑标签

`date_part` `WEEKNUM` `WEEK` `MySQL UDF` `_run_date_part_extraction` `direct_col_map` `源字段回退`



---



#### 问题演进（三轮修复）



**第一轮：修改 SQL 表达式**



用户要求将提取年月周季度的 SQL 改为：

```

年 = YEAR(提取字段)

月 = MONTH(提取字段)

周 = WEEKNUM(提取字段, 2)

季度 = QUARTER(提取字段)

```



原始代码使用 `CAST(... AS UNSIGNED)` 包装且周使用 `WEEK(,3)`。按用户要求修改为去掉 CAST、周改为 `WEEKNUM(,2)`。



**第二轮：批量刷新报错 `FUNCTION WEEKNUM does not exist`**



MySQL 原生不支持 WEEKNUM。`WEEKNUM` 是 Excel 函数，只在 pandas 公式引擎中有对应实现。MySQL 只有 `WEEK(date, mode)`。



公式列 `=YEAR(信息发布日期)` 能正常工作的原因：公式走的是 `translate_formula_to_sql()` → `_translate_weeknum()` 路径，该函数将 Excel `WEEKNUM(date, 2)` 翻译为 MySQL `WEEK(date, 3)`。但日期成分列（`computation_type='date_part'`）走的是 `_build_select()` 中的 `func_map`，直接生成 SQL 函数名，绕过了翻译层。



**修复**：将周的 SQL 表达式回退为 `WEEK(field, 3)`（MySQL 原生），与 `_translate_weeknum` 的翻译结果保持一致。



**第三轮：源字段查找范围不足**



即使 SQL 表达式正确，日期成分列在批量刷新中仍生成为 `NULL`。



**根因**：`_build_select()` 仅在 `direct_col_map` 中查找日期成分的源字段（`date_part_source_field`）。`direct_col_map` 只包含**报表已选列**（即 `col_infos` 中 `is_date_part=False` 且 `is_aggregate=False` 的条目）。如果源字段（如"信息发布日期"）不在报表已选列中，则 `src_info = (None, None)` → `sql_expr = "NULL"`。



公式列为什么不受影响：`translate_formula_to_sql()` 在 `_field_labels`（所有 JOIN 表的所有字段）中搜索，不限于报表已选列。



**修复**：

1. `direct_col_map` 未命中时增加回退搜索：遍历 `self._aliases` 对应的所有源表，在 `_field_labels` 中按字段标签（中文名）或字段键（API 名）匹配，找到后解析 alias 和 quoted_field

2. 此前 `_run_date_part_extraction()` 方法已实现但从未被调用——在 `refresh()` Phase 2.6 新增调用，作为 pandas 兜底
